## 2. 协同编辑场景技术需求分析

### 2.1 CRDT / Yjs 需求

协同编辑的核心挑战是**冲突解决**。Yjs 作为 CRDT 实现提供了：

- **无冲突合并**：多用户同时编辑同一位置，Yjs 自动解决冲突
- **离线支持**：本地状态可离线修改，上线后自动同步
- **增量更新**：只传输变化部分，带宽效率高

**当前状态**：已正确集成 `@tiptap/extension-collaboration` + `@hocuspocus/provider` + `yjs`，技术选型合理。

### 2.2 WebSocket 需求

协同编辑需要**持久连接**支持：

| 需求         | 说明                | 当前实现                |
| ------------ | ------------------- | ----------------------- |
| 实时同步     | 编辑操作毫秒级广播  | Hocuspocus WebSocket ✅ |
| 用户在线状态 | 显示谁在编辑        | 部分实现                |
| 连接恢复     | 断线重连 + 状态恢复 | Hocuspocus 内置 ✅      |
| 水平扩展     | 多实例间状态同步    | Redis 未集成 ❌         |

### 2.3 版本回溯需求

版本回溯需要：

1. **快照存储**：定期保存 Yjs 文档状态快照
2. **差异计算**：版本间的变更对比
3. **回滚操作**：将文档恢复到指定版本
4. **分支管理**：从历史版本创建分支

**当前状态**：Prisma Schema 设计了 `DocumentVersion` 和 `DocumentEdit` 模型，但业务逻辑未实现。

### 2.4 性能要求

| 场景         | 指标                   | 目标     |
| ------------ | ---------------------- | -------- |
| 操作同步延迟 | 用户编辑到其他用户可见 | < 100ms  |
| 文档加载     | 打开文档到可编辑       | < 500ms  |
| 版本快照     | 创建快照不影响编辑     | 异步后台 |
| 并发用户     | 单文档同时编辑         | 50+      |

## 3. Next.js + NestJS 架构合理性评估

### 3.1 架构优势

1. **关注点分离**：前端渲染 (Next.js) 与后端逻辑 (NestJS) 完全解耦
2. **独立部署**：前后端可独立扩缩容
3. **NestJS 适合 WebSocket**：模块化的 Gateway 模式，与 Hocuspocus 集成方便
4. **灵活性**：后端可以同时服务 Web、移动端、桌面端

### 3.2 过度工程化分析

#### 问题 1：Next.js 仅作为 SPA 脚手架

```
当前状态：
- 所有页面都是 'use client'
- 零 RSC (React Server Components) 使用
- 零 SSR (Server-Side Rendering) 利用
- API Routes 未使用（所有 API 走 NestJS）
```

**结论**：Next.js 的核心价值（SSR、RSC、ISR、API Routes）完全未被利用，等同于使用了一个更重的 Create React App。

#### 问题 2：三端口架构复杂度

```
前端 :3000 → 后端 API :3001 → 数据库
               ↓
          WebSocket :3002
```

开发时需要同时启动 3 个服务，部署时需要 3 个端口管理。对于当前阶段的项目，这种分离增加了不必要的复杂度。

#### 问题 3：双框架维护成本

- 两套 TypeScript 配置
- 两套构建流程
- 两套依赖管理
- 类型共享困难
- 部署流程复杂

### 3.3 适用场景评估

**Next + Nest 适合的场景**：

- 大规模企业应用（多团队协作）
- 后端需要服务多种客户端（Web/移动/桌面/IoT）
- 已有 NestJS 后端团队和基础设施
- 微服务架构，后端需要独立演进

**Collab Editor 的实际场景**：

- 单体 Web 应用
- 核心功能是文档编辑（前端为主）
- 后端主要是 WebSocket 中转 + 数据持久化
- 团队规模小，维护双框架成本高

## 4. 替代方案对比

### 方案 A：保持 Next.js + NestJS（现状）

```
架构：Next.js (SPA) ←→ NestJS API ←→ PostgreSQL
                      ←→ Hocuspocus WebSocket
```

| 维度       | 评分       | 说明           |
| ---------- | ---------- | -------------- |
| 开发效率   | ⭐⭐       | 双框架维护     |
| 部署复杂度 | ⭐⭐       | 三端口         |
| 扩展性     | ⭐⭐⭐⭐   | 前后端独立扩容 |
| 当前适用性 | ⭐⭐       | 过度工程化     |
| 迁移成本   | ⭐⭐⭐⭐⭐ | 无需迁移       |

### 方案 B：纯 Next.js + Hocuspocus

```
架构：Next.js (RSC + API Routes) ←→ PostgreSQL
      自定义 Hocuspocus Server    ←→ Redis
```

| 维度       | 评分     | 说明                          |
| ---------- | -------- | ----------------------------- |
| 开发效率   | ⭐⭐⭐⭐ | 单一代码库，共享类型          |
| 部署复杂度 | ⭐⭐⭐⭐ | 减少端口，可部署到 Vercel     |
| 扩展性     | ⭐⭐⭐   | WebSocket 需要 custom server  |
| 当前适用性 | ⭐⭐⭐⭐ | 最适合项目规模                |
| 迁移成本   | ⭐⭐     | 需要重写后端逻辑到 API Routes |

**实现要点**：

- Next.js App Router API Routes 替代 NestJS controllers
- 自定义 Hocuspocus Server 作为独立进程
- Server Actions 处理表单和数据变更
- RSC 实现文档列表等页面的首屏渲染
- 共享 TypeScript 类型（同一代码库）

### 方案 C：纯 NestJS + Vite SPA

```
架构：NestJS (API + 静态文件服务) ←→ PostgreSQL
      Hocuspocus Gateway           ←→ Redis
      Vite 前端 (build 输出到 NestJS static)
```

| 维度       | 评分     | 说明                           |
| ---------- | -------- | ------------------------------ |
| 开发效率   | ⭐⭐⭐   | 仍需维护两套配置               |
| 部署复杂度 | ⭐⭐⭐⭐ | 单一服务                       |
| 扩展性     | ⭐⭐⭐⭐ | 后端驱动                       |
| 当前适用性 | ⭐⭐⭐   | 适合后端为主的项目             |
| 迁移成本   | ⭐⭐⭐   | 前端需要从 Next.js 迁移到 Vite |

## 5. 综合建议

### 5.1 短期建议（当前阶段）

**保持现有架构，但优化使用方式**：

1. **利用 Next.js 能力**：
    - 将文档列表页改为 RSC（Server Component），直接查询后端 API
    - 使用 Server Actions 处理表单提交（注册/登录）
    - 利用 `loading.tsx` 和 `error.tsx` 实现 Streaming UI

2. **统一端口**：
    - 将 Hocuspocus WebSocket 集成到 NestJS 主服务（去除 :3002）
    - 减少一个进程管理

3. **共享类型**：
    - 创建 `packages/shared` 包（monorepo 内）
    - 前后端共享 DTO、接口、枚举

4. **认证改进**：
    - 改用 HttpOnly cookie 存储 JWT
    - 实现 csrf token 机制

### 5.2 中期建议（功能完善阶段）

**评估迁移到方案 B（纯 Next.js）的可行性**：

- 当前后端仅 ~900 行代码，迁移成本可控
- NestJS 的模块化优势在当前规模下不明显
- Hocuspocus 可作为独立进程与 Next.js 并行运行
- 利用 RSC 可以显著改善首屏加载性能

### 5.3 长期建议（扩展阶段）

**如果需要以下场景，保持/回归 NestJS**：

- 多客户端支持（移动端 App、桌面端 Electron）
- 微服务拆分需求
- 复杂的后端业务逻辑（审批流、权限体系、审计日志）
- WebSocket 集群（需要 Redis pub/sub）

### 5.4 技术决策矩阵

| 决策点           | 当前建议 | 触发条件                    |
| ---------------- | -------- | --------------------------- |
| 保持 Next+Nest   | ✅       | 团队 > 3人 或 多客户端需求  |
| 迁移到纯 Next.js | 考虑     | 团队 ≤ 3人 且 仅 Web 客户端 |
| 引入 Redis       | ✅ 必须  | 多人实时协作必需            |
| 共享类型包       | ✅ 必须  | 任何阶段都应做              |
| HttpOnly Cookie  | ✅ 必须  | 安全性要求                  |
| 移除 :3002 端口  | ✅ 建议  | 简化开发体验                |

## 6. 结论

当前 Next.js + NestJS 双框架架构**技术上可行但过度工程化**。对于协同文档编辑器这个产品形态：

1. **核心瓶颈不在架构选型，而在功能实现**：文档 CRUD、版本回溯、协作体验都未完成
2. **建议优先完成功能，而非重构架构**：保持现有架构，利用好 Next.js 的能力（RSC、Server Actions）
3. **Redis 集成是必须的**：多人实时协作场景下，没有 Redis 无法水平扩展 Hocuspocus
4. **如果团队规模小（≤ 3人），纯 Next.js 方案更高效**：减少维护成本，共享类型，简化部署

**最终建议**：短期保持现有架构，优先完成功能实现 + Redis 集成 + 类型共享。在产品验证阶段后，根据团队规模和客户端需求决定是否简化为纯 Next.js 架构。
